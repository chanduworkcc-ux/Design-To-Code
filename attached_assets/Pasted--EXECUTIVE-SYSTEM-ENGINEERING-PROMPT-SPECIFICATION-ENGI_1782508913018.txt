## SECTION 1: USER ONBOARDING, HIGH-SECURITY VALIDATION, AND GATEKEEPING

### A. Strict Onboarding Payload Validation
When a new user initiates the registration process via `POST /api/v1/auth/signup`, you must enforce a strict schema validation check. The request packet must be rejected with a `400 Bad Request` error if any of the following fields are missing, blank, or structurally malformed:
1. `first_name`: Only alphabet characters (`A-Z`, `a-z`). Minimum 2, maximum 50 characters. No numbers, symbols, or trailing whitespaces.
2. `mobile_number`: Numeric string only. Must strictly conform to International E.164 phone numbering standards. 
3. `whatsapp_number`: Numeric string conforming to E.164. The UI must feature a checkbox labeled *"Same as Mobile Number"* which auto-populates this field when toggled.
4. `email_id`: RFC 5322 standard syntax validation (`local-part@domain.extension`).
5. `password`: Minimum 8 characters, maximum 128 characters. Must contain at least one uppercase letter, one lowercase letter, one numeric digit, and one special character.
6. `confirm_password`: Must match the text input string of `password` bit-by-bit.

### B. Anti-Fraud & Deep Verification Suite (Fake Account Prevention)
To eliminate fake, automated, or bot accounts, you must execute validation pipelines prior to inserting data into the database:
* **Disposable Email Blocklist Routing:** Extract the domain portion from `email_id`. Cross-reference it against an active, real-time updated cache of disposable, fake, and temporary email domains. If a match occurs, instantly terminate the process and return: `INVALID_EMAIL_DOMAIN: "Placeholder or temporary email addresses are strictly prohibited."`
* **Mandatory Two-Factor OTP Firewall:** After field checks pass, the account state is marked as `UNVERIFIED_PENDING`. Generate a highly secure, time-sensitive 6-digit One-Time Password (OTP) via your integrated telephony gateway (SMS/WhatsApp API) to the provided `mobile_number`. The system must lock out the registration process until the user inputs the matching OTP within a 5-minute expiry window. No account can be initialized in the database without this successful match.

### C. Algorithmic Gatekeeping & Approval Matrix
Once an account passes validation, evaluate the registration based on the target customer's historical footprint and the global administrative configuration toggle (`registration_approval_mode`):
* **IF `user_segment` == "OLD_CUSTOMER":** Bypass approval gates entirely. Update database status to `status: ACTIVE` and grant immediate access.
* **IF `user_segment` == "NEW_CUSTOMER":** 
  * **Evaluation under MANUAL mode:** Place account into `status: PENDING_ADMIN_REVIEW`. Instantly freeze the user application screen with a non-dismissible security splash screen displaying: *"Welcome! Your registration request has been submitted to the administration team. Access will be granted automatically once your profile is manually reviewed and approved."*
  * **Evaluation under AUTOMATIC mode:** Instantly transition the account to `status: ACTIVE` and generate their authentication token session.

## SECTION 2: BULLETPROOF CHECKOUT & DATA CAPTURE INTEGRITY

### A. Mandatory Shipping & Billing Address Schema
The "Place Order" CTA button inside the shopping cart interface must remain physically unclickable and locked behind a form validation layer. The system must collect and completely verify the following parameters:
* `shipping_recipient_name`: Full legal first and last name of the person accepting delivery.
* `shipping_mobile_number`: Verified E.164 phone string for delivery personnel contact.
* `shipping_pincode`: Strict numeric string matching verified regional postal code databases.
* `shipping_state`: Standard standardized state/province string.
* `shipping_landmark`: A prominent, descriptive spatial point of reference to assist logistics tracking.
* `shipping_full_address`: Granular house number, flat/apartment name, street name, and sector information.

### B. Immediate Order Initialization Logging
The millisecond a user fulfills the payment gateway handoff or confirms Cash on Delivery (COD):
1. Write a permanent, immutable ledger file to the server database.
2. Initialize parameters: `order_id` (UUIDv4 standard), `order_status: CREATED`.
3. Auto-inject a system-wide date-time timestamp: `created_at_timestamp` (Format: `YYYY-MM-DD HH:MM:SS` tracking native local user timezone).
4. **UI Event Driver:** Instantly push a live reactive state change to the customer application interface. This must visually render the initial milestone of the horizontal tracking pipeline.
