Build log error: `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH  Cannot proceed with the frozen installation. The current "overrides" configuration doesn't match the value found in the lockfile` followed by `process "pnpm install --frozen-lockfile --prefer-offline" did not complete successfully: exit code: 1`.
SNAPSHOT_CODE succeeded; BUILD_IMAGE failed at the dependency install step before any build command ran.
There is no railway.json, railway.toml, or railpack.json; the install command is Railpack's default frozen install for a pnpm repo and cannot be changed via a Railway config file.
The same error occurred in the previous deployment (2e7f99f2, commit bec5e20), confirming this is pre-existing lockfile drift, not caused by the latest commit.

## Diagnosis
pnpm 9 stores overrides and catalog entries in pnpm-workspace.yaml and serializes the resolved values into pnpm-lock.yaml. A frozen install fails if those serialized values no longer match the workspace config.
Comparing the two files on branch HEAD: pnpm-workspace.yaml catalog includes `'@replit/vite-plugin-dev-banner': ^0.1.1`, but the lockfile's `catalogs.default` section does not contain `@replit/vite-plugin-dev-banner`. This confirms pnpm-lock.yaml was not regenerated after pnpm-workspace.yaml was edited.
The error names "overrides" because pnpm validates the entire workspace-level config block (overrides + catalogs) as a unit under frozen install. The lockfile is stale and must be regenerated so its overrides and catalogs sections match pnpm-workspace.yaml.
The branch HEAD is unchanged since the failed deployment (branch-HEAD diff is empty), so the issue is still present.

## Fix
1. On a local checkout of branch `main`, run: `pnpm install --no-frozen-lockfile`
2. This updates the root `pnpm-lock.yaml` so its `catalogs.default` and `overrides` sections match the current `pnpm-workspace.yaml` (adding the missing `@replit/vite-plugin-dev-banner` catalog entry and reconciling any other drift).
3. Commit only the regenerated `pnpm-lock.yaml` and push to `main`.
4. This resolves the failure because Railpack's `pnpm install --frozen-lockfile` will then find a lockfile whose overrides/catalog configuration matches `pnpm-workspace.yaml`, allowing the frozen install to proceed.

Do not disable --frozen-lockfile in CI/Railpack as a workaround — that defeats the lockfile integrity check. The correct fix is committing an in-sync lockfile.

Note: this cannot be auto-applied by the code agent because it requires running pnpm to regenerate the lockfile; hand-editing pnpm-lock.yaml is not reliable.
