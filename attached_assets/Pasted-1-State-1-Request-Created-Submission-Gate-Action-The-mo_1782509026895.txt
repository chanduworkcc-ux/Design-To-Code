1. **State 1: Request Created (Submission Gate)**
   * **Action:** The moment a user submits a withdrawal execution request, create a secure transactional instance (`withdrawal_id`).
   * **Data Capture:** Log the dynamic data block: `withdrawal_created_date` and `withdrawal_created_time` (Format: `YYYY-MM-DD HH:MM:SS`).
   * **UI Action:** Display a success toast: *"Your withdrawal request has been sent to the admin successfully."* The first milestone marker on the wallet history page lights up instantly with the recorded creation timestamp.
2. **State 2: Admin Accepted & Admitted (Processing Boundary)**
   * **Action:** The request routes to the Master Admin Panel. When the administrator clicks the **"Accept & Admit"** control interface:
   * **Mandatory Admin Input Fields:** The admin interface *cannot* process the approval without specifying two explicit variables:
     1. `scheduled_credit_date`: The specific calendar date the funds will be added.
     2. `target_resolution_hours`: The total target execution window expressed as a absolute hourly parameter (e.g., `24`, `48`, or `72` hours).
   * **System Calculation Engine:** Upon execution, the server backend must immediately calculate a strict cryptographic countdown constraint: 
     $$\text{Countdown Target Timestamp} = \text{Admin Acceptance Time} + \text{target\_resolution\_hours}$$
3. **State 3: Funds Delivered to Wallet (Settlement State)**
   * **Action:** Once the calculated duration window lapses or the admin fires the final payout instruction, the server adds the exact specified ledger credit back into `user_wallet_balance`.
   * **UI Action:** Mark the third milestone as successfully complete.

### C. Live Real-Time UI Countdown Clock
* **Dynamic Client-Side Ticker:** While a request rests in **State 2 (Accepted / Admitted)**, the user's wallet UI interface must render a live, high-precision visual countdown digital ticker down to the exact second (Format: `HH:MM:SS`).
* **Calculation Anchor:** This clock reads from the server-synchronized `Countdown Target Timestamp`. It forces a dynamic client-side decrease every 1000 milliseconds, showing the user exactly how many hours and minutes remain before the funds land in their digital wallet.
